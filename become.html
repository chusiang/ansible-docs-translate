Ansible特權提升¶

Ansible可以通過一個已存在的特權加強系統使一個使用者以另外一個使用者的身份去執行任務

Become¶

Ansible 1.9之前主要允許使用者使用sudo和有限的su命令來以不同使用者的身份/許可權遠端登陸執行task,及建立資源. 在1.9版本中’become’取代了之前的sudo/su, 但是sudo/su依舊向後相容. 這個新系統使得我們更加容易的增加其他的特權提升工具, 例如pbrun(Powerbroker), pfexec等.

新指令¶

become
等同於新增 ‘sudo:’ 或 ‘su:’ (指令)去執行task/playbook, 將其設定為 ‘true’/’yes’ 來啟用特權提升
become_user
等同於新增 ‘sudo_user:’ 或 ‘su_user:’ (指令)去執行task/playbook, 設定為擁有所渴望許可權的使用者
become_method
執行task/playbook時的特權提升method, 覆蓋ansible.cfg中的預設配置, 可以設定為’sudo’/’su’/’pbrun’/’pfexec’/’doas’

新的ansible_ 變數¶

在每一個組或host中都允許你作為一項來設定

ansible_become
等同於ansible_sudo或ansible_su, 設定為強制進行特權提升操作
ansible_become_method
用來設定特權提升所使用的方法
ansible_become_user
等同於ansible_sudo或ansible_su_user, 設定通過特權提升後的使用者身份
ansible_become_pass
等同於ansible_sudo_pass或ansible_su_pass, 設定通過特權提升的密碼

新的命令列參數¶

--ask-become-pass
 詢問特權提升方法的密碼
–become,-b
指定執行的become(預設沒有密碼)
--become-method=BECOME_METHOD
 使用的特權提升方式(預設=sudo), 可選的合法值: [ sudo | su | pbrun | pfexec | doas ]
--become-user=BECOME_USER
 指定執行的使用者(預設=root)

sudo和su依舊可以工作¶

舊的playbooks不需要修改, 儘管我們不建議在使用之前的語法, sudo和su之類依然可以工作可是我們還是建議使用become將他們一次性替換. 因為你不能在同一個物件中混合使用這兩種指令, 否則ansible將會引發異常.

如果sudo/su的配置或變數存在become將預設使用這些設定, 但是如果你指定任何一個配置都會將他們覆蓋.

Note

許可權加固方法必須支援連線外掛使用, 如果不支援將會引發警告, 大部分會時候會忽略它,因為這些操作會一直以root使用者的身份去執行(jail, chroot, etc).

Note

特權提升方法不能被重組, 你不能使用’sudo /bin/su -‘來成為一個使用者, 你需要執行的命令必須在能通過sudo或su指令來獲得執行許可權(這同樣使用於pbrun, pfexec或其他方法)

Note

Privilege escalation permissions have to be general, Ansible does not always use a specific command to do something but runs modules (code) from a temporary file name which changes every time. So if you have ‘/sbin/sevice’ or ‘/bin/chmod’ as the allowed commands this will fail with ansible.

Note

許可權加固策略必須通用, Ansible並不是使用一個明確的命令去執行task, 它將通過一個臨時的檔案來執行模組(程式碼), 每執行一次命令這個臨時檔案的檔名就會發生一次變化. 所以如果你有 ‘/sbin/service’ 或 ‘/bin/chmod’ 作為允許的命令, 在ansible中可能會失敗.
